# Linear PM Handoff

`linear` CLI を使って、Linear issue 群を「別のエージェントや後続担当が迷わず復帰できる状態」に整えるスキル。

`linear-pm` がタスク構造の整理を担うのに対し、このスキルは handoff と restartability に特化する。

## 最重要原則

**最終的な基準は「コンテキストを持たない別エージェントが、エントリポイント issue だけで次に読むべき issue と再開位置を把握できるか」である。**

そのために必ず以下を行う:

- 完了済み issue は `Done` に更新する
- 途中 issue は `In Progress` のまま、やったところまでを詳細コメントに残す
- 各途中 issue に「次に何から始めるか」「復帰時に何を読めばよいか」を書く
- 親 issue / エントリポイント issue に短い復帰導線を置く
- subagent に実際に読ませて復帰可能かを検証する
- 検証が弱ければコメントを改善して再検証する

## いつ使うか

次の状況ではこのスキルを優先する:

- セッション終了前に handoff したい
- 複数の Linear issue を進めた後、状態更新と進捗記録をまとめたい
- 「`KEN-202 をロード` だけで復帰できる」ようにしたい
- 次の担当が issue ツリーを追いやすいようにしたい
- subagent で引き継ぎ可能性を検証したい

## Workflow

### 1. 対象 issue を読む

推測で書かない。必ず現在の issue 状態・親子関係・最近のコメントを読む。

```bash
linear issue view KEN-202 --json
linear issue view KEN-217 --json
```

親子や依存が必要なら API で確認する:

```bash
linear api --variable id=<issueId> <<'GQL'
query($id: String!) {
  issue(id: $id) {
    identifier
    title
    state { name type }
    parent { identifier title }
    children { nodes { identifier title state { name type } } }
  }
}
GQL
```

### 2. 状態を正しく揃える

状態更新は handoff の一部であり、後回しにしない。

- 完了したものは `Done`
- 途中のものは `In Progress`
- 未着手は無理に動かさない

```bash
linear issue update KEN-216 --state "Done"
linear issue update KEN-217 --state "In Progress"
```

### 3. 各 issue に進捗コメントを書く

コメントは「やったこと」だけでなく「途中の位置」を残す。短すぎるサマリは禁止。

途中 issue のコメントには最低限これを含める:

- 今回やったこと
- まだ終わっていないこと
- 次に何から始めるか
- 復帰時に先に読むべき issue / コメント
- 触ってはいけない制約

完了 issue のコメントには最低限これを含める:

- 何をもって完了としたか
- 関連 issue への影響
- 次に進むべき issue

途中 issue のコメント例:

```markdown
今回ここまで進めた:
- manifest loader を追加
- platformSpec への変換を追加

未完:
- systemd/timer の標準形はまだ詰めきっていない

復帰手順:
1. まず KEN-202 の最新コメントを読む
2. 次にこの KEN-217 の最新コメントを読む
3. 既存サービスは触らず、platform 側だけ進める
4. 次は systemd/timer の標準形を詰める
```

### 4. エントリポイント issue に復帰導線を置く

親 issue または最上位 issue には、短い導線コメントを必ず残す。

最低限含める:

- どの issue から再開するか
- 何を触ってはいけないか
- 迷ったらどの issue を読めばよいか

短い導線の例:

```markdown
KEN-202 の最新コメントから復帰。次は KEN-217。既存サービスは触らず、platform 側だけ進める。迷ったら KEN-209 と KEN-217 の最新コメントを読む。
```

必要ならエントリポイントには別途、少し長めの復帰メモも置いてよい。ただし短い導線は必ず残す。

### 5. subagent で検証する

**ここがこのスキルのキモ。**

subagent が使える環境では、必ず subagent を使う。コンテキストを持たない前提で以下だけを渡して検証する:

- エントリポイント issue をロードする指示
- 実作業はしないこと
- 復帰に必要な情報を把握できるかを確認すること

subagent の呼び出し手段は環境依存だが、要件は同じ:

- 新しい subagent / explorer を起動する
- 共有済みの会話履歴を最小限にする
- エントリポイント issue 以外は必要になるまで読ませない
- 実作業禁止を明示する

検証で確認したい項目:

- 次に着手すべき issue が分かるか
- 完了済み / 進行中の認識が取れるか
- やってはいけない制約が分かるか
- 実作業前にどの issue / コメントを読めばよいか分かるか
- 迷ったときの戻り先が分かるか

subagent への依頼例:

```text
Linear の KEN-202 だけを起点に、次に何を読むべきかと次の作業が何かを把握してください。実際のタスク実行はしないでください。復帰に足りない情報があれば不足点を指摘してください。
```

subagent が使えない環境では fallback として、次善の検証を行う:

- エントリポイント issue の本文と最新コメントだけを見て復帰メモを一度ゼロから書き出す
- そのメモが対象 issue と次アクションまで辿れているか自分で照合する
- ただしこれは subagent 検証の代替であり、同等ではないと扱う

### 5.5 検証結果を必ず記録する

検証しただけで終わらせない。結果は必ず issue に残す。

- エントリポイント issue に「検証結果」コメントを残す
- 内容は「取れたこと」「不足していたこと」「次に直した点」を簡潔に書く
- 改善後に再検証した場合は、その結果も追記する

検証結果コメントの例:

```markdown
検証結果:
- `KEN-202 をロード` だけで次は KEN-217 だと分かった
- 既存サービスを触らない制約も取れた
- KEN-217 の具体的再開内容は曖昧だった

改善:
- KEN-202 に「次は KEN-217」「KEN-217 の進捗コメントを読め」を追記
```

### 6. 改善ループを回す

subagent 検証で不十分なら、それで終わらせない。足りなかった情報を issue コメントへ戻し、再検証する。

改善対象の典型:

- エントリポイントだけでは次 issue まで辿れない
- 次 issue は分かるが、何をするか曖昧
- 制約が issue コメントに書かれていない
- どのコメントを読むべきか書いていない

改善ループ:

1. subagent の不足指摘を要約する
2. エントリポイント issue に短い導線を追加または修正する
3. 対象 issue に復帰手順を追記する
4. エントリポイント issue に検証結果コメントを残す
5. 必要なら親 issue にも進捗サマリを追記する
5. subagent に再検証させる

**「KEN-XXX をロード」だけで復帰したいなら、その一文だけで次の issue に辿れるまでループする。**

## コメントの書き方

良いコメントの条件:

- 抽象論ではなく、今回やった具体的な事項がある
- 次の着手点が一文で分かる
- 制約が明示されている
- 読む順番が書いてある

悪い例:

```markdown
進めました。次もよろしくお願いします。
```

良い例:

```markdown
完了:
- KEN-216 は manifest 入口と parse/normalize まで完了

進行中:
- KEN-217 は systemd/timer 標準形の詰めが未完

復帰:
- KEN-202 の最新コメントから入り、次に KEN-217 の最新コメントを読む
- 既存サービスは触らず、platform 側だけ進める
```

## 出力の最低ライン

このスキルを使ったターンの終了時には、最低でも以下が揃っている状態にする:

- 完了 issue の状態更新
- 進行中 issue の状態更新
- 完了 issue の完了コメント
- 進行中 issue の途中経過コメント
- エントリポイント issue の短い復帰導線
- subagent による検証結果
- エントリポイント issue への検証結果コメント
- 不十分だった場合の改善反映

## 実務上の注意

- 実作業と handoff 整理を混ぜない。handoff を頼まれたら、勝手に新しい実装作業へ進まない
- issue コメントは次担当向けに書く。自分向けの省略メモにしない
- 「完了」「進行中」は実態に合わせる。見栄えのために Done を増やさない
- subagent には実作業禁止を明示する
- subagent が使えない場合は fallback 検証で妥協し、その旨を明記する
- subagent の検証結果は必ずエントリポイント issue に還元する
